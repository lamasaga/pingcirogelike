## 适用范围

　　本文档描述如何将 **拼词肉鸽**（纯静态 HTML / ES Module，无 Node 后端）部署到 Ubuntu 服务器目录 **`/srv/pingchi`**，并通过公网 IP **`82.156.225.73`** 以小范围 HTTP 服务访问（示例 URL：**`http://82.156.225.73/pingchi/`**）。

　　与 TechVenture（Node + systemd）不同：本项目 **不需要** `npm run build`、**不需要** 为拼词单独建 systemd 服务；更新代码后 **无需重启应用进程**，仅需保证文件已同步（必要时 `nginx reload` 仅在改配置时）。

---

## 首次部署：登录服务器后一次性执行（推荐）

　　在你 **本机能 SSH 登录** 的前提下（例如 `ssh -i 你的密钥 ubuntu@82.156.225.73`），登录后 **整段复制执行** 下列命令即可：先拉代码，再安装 Nginx 片段并 reload。（若仓库里还没有 `deploy/` 目录，请先在本机 `git push` 最新提交后再执行。）

```bash
set -e
REPO_URL="https://github.com/lamasaga/pingcirogelike.git"
TARGET="/srv/pingchi"
sudo mkdir -p "$TARGET"
sudo chown -R ubuntu:ubuntu "$TARGET"
if [ ! -d "$TARGET/.git" ]; then git clone "$REPO_URL" "$TARGET"; else git -C "$TARGET" pull origin main; fi
chmod -R a+rX "$TARGET"
sudo cp "$TARGET/deploy/pingchi-subpath.snippet.conf" /etc/nginx/snippets/pingchi-subpath.conf
```

　　接着在 **处理公网 IP、监听 80 端口** 的那个 `server { ... }` 里加入一行（与现有 `location` 同级）：

```nginx
include /etc/nginx/snippets/pingchi-subpath.conf;
```

　　若不确定改哪个文件，可先执行：

```bash
grep -RIn "listen .*80" /etc/nginx/sites-enabled/
```

　　编辑对应 `.conf`，把 `include` 放进合适的 `server {}` 内，保存后执行：

```bash
sudo nginx -t && sudo systemctl reload nginx
curl -I http://127.0.0.1/pingchi/
```

　　浏览器访问：**`http://82.156.225.73/pingchi/`**。

　　等价方式：仓库内 **`deploy/setup-pingchi.sh`** 仅负责拉代码与权限；Nginx 仍需按上文 `include` 一次。

```bash
chmod -R a+rX /srv/pingchi
```

## Nginx 配置（IP + 子路径 `/pingchi/`）

　　将下列片段并入 **当前处理 `82.156.225.73` 或 `default_server` 的 `server { ... }`**（与现有 `location` 并存即可）。若你已有默认站点文件，一般是 `/etc/nginx/sites-enabled/` 下某个 `.conf`。

```nginx
# 无尾部斜杠时跳转到带斜杠，避免 css/js 相对路径解析到错误路径
location = /pingchi {
    return 301 http://82.156.225.73/pingchi/;
}

location /pingchi/ {
    alias /srv/pingchi/;
    index index.html;
}
```

　　检查并重载：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

　　验收：

```bash
curl -I http://127.0.0.1/pingchi/
curl -I http://82.156.225.73/pingchi/
```

　　期望 `index.html` 返回 **200**；访问 `http://82.156.225.73/pingchi`（无斜杠）应 **301** 到带斜杠地址。

### 与 TechVenture 同机共存时

　　TechVenture 若通过 **域名 + HTTPS** 走向 Node，与本段 **按 IP 访问静态子路径** 可同时存在：注意不要为同一 `listen`/`server_name` 重复定义冲突的 `location /pingchi/`。若 80 端口已有多个 `server` 且都监听同一 IP，请把上述 `location` 放进 **实际接收浏览器访问 IP 请求** 的那一个 `server` 块内。

## 安全与范围说明（小范围服务）

　　使用公网 IP + HTTP 时：**传输未加密**，适合信任网络或小范围演示。若日后绑定域名，可为同一 `server_name` 申请证书并改为 HTTPS，再把上面 `return 301` 里的 URL 改成 `https://`。

　　可通过云厂商安全组将 **80 端口** 限制为指定来源 IP，以缩小暴露面。

## 常见故障

### 1）页面空白、控制台报 css/js 404

　　多为访问了 `http://82.156.225.73/pingchi` **未带尾部斜杠**。请确认已配置 `location = /pingchi { return 301 ... }`。

### 2）403 Forbidden

　　检查 `/srv/pingchi` 及上级目录是否允许 Nginx 用户遍历执行权限（`a+rX`），以及 `alias` 路径是否与磁盘一致。

### 3）502 / 与其它站点混淆

　　确认 `location /pingchi/` 所在 `server` 块正是浏览器访问 IP 时命中的虚拟主机（可用 `curl -H "Host: 82.156.225.73" http://127.0.0.1/pingchi/` 辅助排查）。
