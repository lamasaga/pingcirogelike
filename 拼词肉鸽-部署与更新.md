## 适用范围

　　本文档描述如何将 **拼词肉鸽**（纯静态 HTML / ES Module，无 Node 后端）部署到 Ubuntu 服务器目录 **`/srv/pingchi`**，并通过公网 IP **`82.156.225.73`** 以小范围 HTTP 服务访问（示例 URL：**`http://82.156.225.73/pingchi/`**）。

　　与 TechVenture（Node + systemd）不同：本项目 **不需要** `npm run build`、**不需要** 为拼词单独建 systemd 服务；更新代码后 **无需重启应用进程**，仅需保证文件已同步（必要时 `nginx reload` 仅在改配置时）。

## 服务器无法直连 GitHub：Windows 打包上传（推荐给你当前环境）

　　若服务器上 **`curl`/git 访问 GitHub 443 超时**（出境不稳定），不要在服务器上 `git clone`，改为 **本机打包 → `scp` 上传 → 服务器解压**。

### 方式 A：一键脚本（推荐）

　　在本机 **PowerShell** 中（注意先把路径改成你的仓库路径；密钥默认使用 `%USERPROFILE%\.ssh\alevelinfo_ed25519`，可用参数覆盖）：

```powershell
Set-Location "d:\1XFAwork\CURSORWork\【Gamer】\PINGCHI"
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\deploy\package-and-upload.ps1
```

　　可选参数示例：

```powershell
.\deploy\package-and-upload.ps1 -KeyPath "C:\Users\你的用户\.ssh\你的密钥" -ServerHost "82.156.225.73"
```

　　脚本会用 **Windows 自带的 `tar.exe` 打 `tar.gz`**（保留 **中文路径**，并排除 `.git` / `node_modules` / `.cursor` / `.obsidian`）→ 上传到 **`/tmp/pingchi-deploy.tar.gz`** → 调用 **`deploy/remote-extract.sh`**：清空并解压到 **`/srv/pingchi`**、安装 **`/etc/nginx/snippets/pingchi-subpath.conf`**；若检测到 nginx 里 **尚未** `include` 该文件，会提示你手动加一行（仅需做一次），不会强行改坏现有站点配置。

　　**不要使用 `Compress-Archive` 打 zip 再上传**：在 Linux 上 `unzip` 常出现 **中文目录乱码**（如 `icons/图标` 变成乱码名），且可能提示 **backslashes as path separators**，遗物图片路径会对不上。

### 方式 B：手动命令（便于理解每一步）

　　**本机 PowerShell：**

```powershell
Set-Location "d:\1XFAwork\CURSORWork\【Gamer】\PINGCHI"
$archive = "$env:TEMP\pingchi-deploy.tar.gz"
if (Test-Path $archive) { Remove-Item $archive -Force }
tar.exe -czvf $archive --exclude=.git --exclude=node_modules --exclude=.cursor --exclude=.obsidian -C $PWD .
scp -i "$env:USERPROFILE\.ssh\alevelinfo_ed25519" $archive ubuntu@82.156.225.73:/tmp/pingchi-deploy.tar.gz
scp -i "$env:USERPROFILE\.ssh\alevelinfo_ed25519" ".\deploy\remote-extract.sh" ubuntu@82.156.225.73:/tmp/pingchi-remote-extract.sh
ssh -i "$env:USERPROFILE\.ssh\alevelinfo_ed25519" ubuntu@82.156.225.73 "sed -i 's/\r$//' /tmp/pingchi-remote-extract.sh && bash /tmp/pingchi-remote-extract.sh /tmp/pingchi-deploy.tar.gz"
```

　　**nginx `include`（若脚本提示尚未配置）**：在监听 **80** 的 `server {}` 内增加一行 `include /etc/nginx/snippets/pingchi-subpath.conf;`，再执行 `sudo nginx -t && sudo systemctl reload nginx`。详见下文「Nginx 配置」与「首次部署」中的说明。

## 首次部署：登录服务器后一次性执行（推荐）

　　在你 **本机能 SSH 登录** 的前提下（例如 `ssh -i 你的密钥 ubuntu@82.156.225.73`），登录后 **整段复制执行** 下列命令即可：先拉代码，再安装 Nginx 片段并 reload。（若仓库里还没有 `deploy/` 目录，请先在本机 `git push` 最新提交后再执行。）

```bash
set -e
REPO_URL="https://github.com/lamasaga/pingcirogelike.git"
TARGET="/srv/pingchi"
sudo mkdir -p "$TARGET"
sudo chown -R ubuntu:ubuntu "$TARGET"
if [ ! -d "$TARGET/.git" ]; then git clone "$REPO_URL" "$TARGET"; else git -C "$TARGET" pull origin main; fi
chmod -R a+rX "$TARGET"
sudo cp "$TARGET/deploy/pingchi-subpath.snippet.conf" /etc/nginx/snippets/pingchi-subpath.conf
```

　　接着在 **处理公网 IP、监听 80 端口** 的那个 `server { ... }` 里加入一行（与现有 `location` 同级）：

```nginx
include /etc/nginx/snippets/pingchi-subpath.conf;
```

　　若不确定改哪个文件，可先执行：

```bash
grep -RIn "listen .*80" /etc/nginx/sites-enabled/
```

　　编辑对应 `.conf`，把 `include` 放进合适的 `server {}` 内，保存后执行：

```bash
sudo nginx -t && sudo systemctl reload nginx
curl -I http://127.0.0.1/pingchi/
```

　　浏览器访问：**`http://82.156.225.73/pingchi/`**。

　　等价方式：仓库内 **`deploy/setup-pingchi.sh`** 仅负责拉代码与权限；Nginx 仍需按上文 `include` 一次。

## Nginx 配置（IP + 子路径 `/pingchi/`）

　　将下列片段并入 **当前处理 `82.156.225.73` 或 `default_server` 的 `server { ... }`**（与现有 `location` 并存即可）。若你已有默认站点文件，一般是 `/etc/nginx/sites-enabled/` 下某个 `.conf`。

```nginx
# 无尾部斜杠时跳转到带斜杠，避免 css/js 相对路径解析到错误路径
location = /pingchi {
    return 301 http://82.156.225.73/pingchi/;
}

location /pingchi/ {
    alias /srv/pingchi/;
    index index.html;
}
```

　　检查并重载：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

　　验收：

```bash
curl -I http://127.0.0.1/pingchi/
curl -I http://82.156.225.73/pingchi/
```

　　期望 `index.html` 返回 **200**；访问 `http://82.156.225.73/pingchi`（无斜杠）应 **301** 到带斜杠地址。

### 与 TechVenture 同机共存时

　　TechVenture 若通过 **域名 + HTTPS** 走向 Node，与本段 **按 IP 访问静态子路径** 可同时存在：注意不要为同一 `listen`/`server_name` 重复定义冲突的 `location /pingchi/`。若 80 端口已有多个 `server` 且都监听同一 IP，请把上述 `location` 放进 **实际接收浏览器访问 IP 请求** 的那一个 `server` 块内。

## 安全与范围说明（小范围服务）

　　使用公网 IP + HTTP 时：**传输未加密**，适合信任网络或小范围演示。若日后绑定域名，可为同一 `server_name` 申请证书并改为 HTTPS，再把上面 `return 301` 里的 URL 改成 `https://`。

　　可通过云厂商安全组将 **80 端口** 限制为指定来源 IP，以缩小暴露面。

## 常见故障

### 1）页面空白、控制台报 css/js 404

　　多为访问了 `http://82.156.225.73/pingchi` **未带尾部斜杠**。请确认已配置 `location = /pingchi { return 301 ... }`。

### 2）403 Forbidden

　　检查 `/srv/pingchi` 及上级目录是否允许 Nginx 用户遍历执行权限（`a+rX`），以及 `alias` 路径是否与磁盘一致。

### 3）502 / 与其它站点混淆

　　确认 `location /pingchi/` 所在 `server` 块正是浏览器访问 IP 时命中的虚拟主机（可用 `curl -H "Host: 82.156.225.73" http://127.0.0.1/pingchi/` 辅助排查）。

### 4）本机 `scp`/`ssh` 长时间停在「Upload …」

　　多为网络抖动或密钥认证卡住；确认 `-KeyPath` 对应的本机密钥已在服务器 `authorized_keys` 中。可在本机单独测：`scp -i 密钥 小文件 ubuntu@IP:/tmp/`。上传较大 `tar.gz` 时链路慢属正常现象。

### 5）远程执行脚本报 `set: pipefail: invalid option name`

　　多为 **Windows 保存的 `.sh` 使用 CRLF 换行**，上传到 Linux 后行尾 `\r` 破坏了 `set -euo pipefail`。处理：在服务器执行一次 `sed -i 's/\r$//' /tmp/pingchi-remote-extract.sh` 后再 `bash`；或重新拉取已设置 **`*.sh eol=lf`** 的仓库后再上传。一键脚本已默认带 `sed` 修正步骤。

### 6）解压后中文文件名/目录乱码、`unzip` 提示反斜杠路径

　　多为使用了 **`Compress-Archive` + zip**。请改用 **`tar.gz` 流程**（一键脚本或上文方式 B），删除服务器上 `/srv/pingchi` 后重新部署一次。

### 7）`ssh -i` 能登录，但脚本里 `scp` 仍问密码 / Permission denied

　　常见原因是 **`scp` 用的不是系统自带的 OpenSSH**（PATH 里别的 `scp` 可能不按 `-i` 工作），或客户端尝试了多把密钥后退回密码。一键脚本已改为优先使用 **`%SystemRoot%\System32\OpenSSH\scp.exe`**，并加上 **`IdentitiesOnly=yes`**。若仍异常，在本机执行 **`where.exe scp`**，确认第一个路径是否为上述 OpenSSH；也可手动测：`scp -o IdentitiesOnly=yes -i "私钥完整路径" 某小文件 ubuntu@IP:/tmp/`。
