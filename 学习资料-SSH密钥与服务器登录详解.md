## 这份文档讲什么

　　你用 **SSH** 从 Windows 登录 Ubuntu、用 **`scp` 上传部署包** 时，常会碰到 **密钥路径**、**密码提示**、**Permission denied**。这些几乎都和对 **「谁持有私钥、服务器保存了哪把公钥」** 的理解有关。下文按 **概念 → 第一次配置 → 日常使用 → 排错与轮换** 说明；不涉及高深密码学公式，只保留运维必需的认识。

---

## 1. 一对钥匙分别是什么

| 名称 | 文件常见形态 | 谁能复制、谁能泄露 |
|------|----------------|-------------------|
| **私钥（private key）** | 本机 `id_ed25519`、`*.pem`、无后缀私钥文件 | **只有你这台可信机器该保留**；**绝不**发给他人、**绝不**提交到 Git、**绝不**贴到聊天或邮件 |
| **公钥（public key）** | `id_ed25519.pub`、云控制台「粘贴公钥」那一整行 | **可以**交给服务器、云平台、Git 托管（GitHub Deploy keys 等）；泄露公钥本身危害远小于私钥泄露 |

　　登录验证的直觉模型：**你用私钥签一段证明 → 服务器用事先保存的公钥验证「确实是这把钥匙的主人」**。没有私钥的人无法冒充；服务器上没有对应公钥时，你就登不进去（除非另开密码登录且服务端允许）。

## 2. 何时「生成」密钥对

　　典型时机：

- **第一次**要从某台 **个人电脑** 稳定登录某云主机，且希望 **不用每次输服务器密码**；
- **换新电脑** 后，在新电脑上为新用途生成新密钥（或安全迁移旧私钥，见下文）；
- **区分用途**：例如「登录服务器的密钥」与「访问 GitHub 的密钥」分开（可选，但更清晰）。

　　生成命令（在 **本机** PowerShell 或 Git Bash 里执行一次即可）：

```powershell
ssh-keygen -t ed25519 -C "你的备注例如笔记本Win11"
```

　　提示保存路径时可直接回车（默认 `C:\Users\你的用户\.ssh\id_ed25519`），可选设置 **passphrase**（私钥额外口令；更安全，但每次使用前可能要输入或由 ssh-agent 托管）。

## 3. 公钥、私钥各自「交给谁」、在什么时机交

### 3.1 私钥：永远不「交给」服务器

　　**私钥只留在客户端**（你的 Windows）。部署脚本里的 **`-i "...\xxxxx"`** 指向的就是私钥路径。

### 3.2 公钥：在「你希望自己能登录服务器」时交给服务器

　　时机通常是：**买好云主机、装好系统之后**，第一次配置远程登录。

　　常见几种落地方式（选一种即可）：

1. **创建实例时在控制台粘贴公钥**  
   　　云平台把公钥写入镜像里的 **`ubuntu`（或 root）用户的 `authorized_keys`**，实例启动后你就能用对应私钥登录。

2. **实例已存在：手动追加公钥**  
   　　你能用 **控制台 VNC / 临时密码** 先进系统后，把本机 **`xxx.pub` 文件内容**追加到服务器：

   　　`/home/ubuntu/.ssh/authorized_keys`

   　　（注意权限：`~/.ssh` 一般为 `700`，`authorized_keys` 为 `600`。）

3. **`ssh-copy-id`**（Linux/Mac 常见；Windows 可装 Git Bash 或用 PowerShell 手动复制）

　　**只有你（或运维）把公钥放进 `authorized_keys` 的那一刻起**，持有匹配私钥的你才能 **免服务器密码** 登录（若服务端关闭了密码登录，这是唯一途径）。

## 4. 和你当前工作的对应关系

| 场景 | 你手里的 | 服务器上的 |
|------|-----------|------------|
| `ssh -i C:\...\alevelinfo_ed25519 ubuntu@IP` | **私钥** `alevelinfo_ed25519` | **`authorized_keys` 里应有与该私钥配对的公钥那一行** |
| `.\deploy\package-and-upload.ps1 -KeyPath "..."` | 同上（脚本内部调用 `scp`/`ssh`） | 同上 |

　　若 **`authorized_keys` 里是别的公钥**（例如当初绑的是腾讯云网页生成的 RSA），而你本地用的是 **`alevelinfo_ed25519`**，就会出现：**明明写了 `-i`，仍被要求输入密码 / Permission denied**——这不是命令写错「路径格式」，而是 **钥匙和服务端登记的不是一对**。

## 5. 日常管理建议（可持续的做法）

### 5.1 备份谁、不备份谁

- **备份**：私钥文件的 **加密备份**（例如加密 U 盘、密码管理器附件）；若用了 passphrase，备份时记下口令策略。  
- **不必同等紧张**：公钥 `.pub` 可多份留存；服务器上的 `authorized_keys` 建议定期看一眼有没有陌生行。

### 5.2 一台电脑多把密钥

　　可以在 **`C:\Users\你\.ssh\config`** 里为不同主机指定不同 **`IdentityFile`**，避免每次手写一长串 `-i`：

```ssh-config
Host mycloud
    HostName 82.156.225.73
    User ubuntu
    IdentityFile C:/Users/MECHREVO/.ssh/alevelinfo_ed25519
    IdentitiesOnly yes
```

　　之后：`ssh mycloud` 即可。

### 5.3 Windows 私钥权限

　　权限过宽时，OpenSSH 可能 **拒绝使用私钥**，表现为又去走密码。可对单个私钥收紧 ACL（略，详见部署排错对话或微软文档）。

### 5.4 与 GitHub 的密钥区分开（概念）

- **登录服务器**：公钥进服务器的 **`authorized_keys`**。  
- **GitHub 读私有仓库 / Deploy key**：公钥进 **GitHub 仓库或账号设置**，私钥在 CI 或服务器上。

　　二者用途不同，不要混用同一文件路径却在错误的地方登记。

## 6. 轮换、作废与泄漏应急

- **怀疑私钥泄露**：立刻在服务器上 **删除 `authorized_keys` 中对应公钥那一行**，换新生成的密钥对并登记新公钥。  
- **员工离职 / 设备丢失**：同上，从 **`authorized_keys`** 移除旧公钥。  
- **轮换间隔**：无固定国标；敏感环境可每年或按单位规范更换。

---

## 7. Passphrase 与 ssh-agent（可选进阶）

　　生成密钥时若设置了 **passphrase**，每次使用私钥可能要多输一次口令。**ssh-agent** 可在一次解锁后，本会话内免重复输入。Windows 上常用「OpenSSH Authentication Agent」服务或内置集成；细节可查微软 OpenSSH 文档。

## 8. 常见问题（对照你遇到的现象）

| 现象 | 常见原因 |
|------|-----------|
| 仍提示 `password:` | 私钥未被使用（路径错、`$key` 为空）、或服务端不接受该私钥 |
| `Permission denied (publickey)` | 公钥未进 `authorized_keys`，或与当前私钥不匹配 |
| `Permission denied (publickey,password)` | 密钥失败后又尝试密码；密码错或服务端禁密码 |
| 脚本里写了 `-KeyPath` 仍问密码 | 路径不存在、权限过宽导致密钥被忽略、或服务端无对应公钥 |

　　调试可用：`ssh -vvv -o IdentitiesOnly=yes -i "私钥路径" user@host`，从日志里看 **Offering public key** 是否出现、是否 **Accepted**。

---

## 9. 和「服务器防火墙 / Nginx」的关系

　　**密钥只解决「你是谁、能否登录」**。**安全组放行 22 端口**解决「网络能否到达 SSH 服务」。二者都要满足：**22 放行 + 密钥正确**，你才能 `ssh` / `scp`。
