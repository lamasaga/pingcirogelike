# 使用策略映射替代长链 if-else：可扩展的角色技能设计模式

在处理多角色、多行为（如技能、AI动作、状态效果等）的系统时，应避免使用冗长的 `if-else` 或 `switch` 语句硬编码每个分支。这种写法在角色数量增长时会导致代码臃肿、难以维护且违反开闭原则。

**推荐做法**：采用 **策略映射（Strategy Mapping）** 模式——将每种行为封装为独立函数（或对象），并通过一个配置驱动的映射表（如对象或 Map）动态调用。

## 1. 为什么这种模式更好？

*   **可读性**：主循环（如 `tryCastUltimate`）变得极其简洁，只负责“调度”，不负责“实现”。
*   **可维护性**：修改某个角色的技能只需修改对应的策略函数，不会误伤其他角色。
*   **可扩展性**：新增角色时，只需在映射表中添加一项，完全不需要触碰主逻辑代码。
*   **解耦**：技能逻辑可以拆分到单独的文件中，避免 `Game.js` 变成几千行的“上帝对象”。

## 2. 代码重构示例

### 重构前 (当前 `Game.js` 的写法)

```javascript
tryCastUltimate() {
    // ... 通用前置逻辑 ...
    
    if (this.config.hero.id === 0) {
        // 几十行 角色0 的逻辑
        // ...
    }
    else if (this.config.hero.id === 1) {
        // 几十行 角色1 的逻辑
        // ...
    }
    else if (this.config.hero.id === 2) {
        // 几十行 角色2 的逻辑
        // ...
    }
    // 如果有 20 个角色，这里将变成噩梦
}
```

### 重构后 (策略模式)

#### A. 定义技能策略 (可以单独放在 `SkillStrategies.js`)

```javascript
// 技能实现细节被隔离在这里
export const SkillStrategies = {
    // 角色 0: Time Freeze
    0: (game) => {
        game.state.isTimeFrozen = true;
        AudioManager.stopBgm();
        
        // 计算持续时间
        const skillDuration = 12000 * game.config.stats.skillDurationMult;
        
        // UI 更新
        const statusEl = document.getElementById('skill-status');
        if(statusEl) { statusEl.style.display = 'block'; statusEl.innerText = 'TIME FROZEN'; }

        // 定时恢复
        setTimeout(() => {
            game.state.isTimeFrozen = false;
            if (game.state.isPlaying) {
                 const bgmKey = `battle_${game.config.zone.id}`;
                 AudioManager.playBgm(bgmKey);
            }
            if(statusEl) statusEl.style.display = 'none';
        }, skillDuration);
    },

    // 角色 1: Explosion Charges
    1: (game) => {
        const heroSave = DataManager.getHeroSave(game.config.hero.id);
        // 具体的数值逻辑也在这里维护
        const charges = 8 + (heroSave.skillLevel - 1) * 2;
        
        game.state.aoeCharges = charges;
        game.input.inputBuffer = ''; 
    },

    // 角色 2: Lightning Chain
    2: (game) => {
        const heroSave = DataManager.getHeroSave(game.config.hero.id);
        const targetCount = 15 + (heroSave.skillLevel - 1) * 5;
        
        const center = game.playerPos;
        const targets = [...game.entities.monsters]
            .map(m => ({ m, dist: Math.hypot(m.x - center.x, m.y - center.y) }))
            .sort((a, b) => a.dist - b.dist)
            .slice(0, targetCount);

        targets.forEach(t => {
            const idx = game.entities.monsters.indexOf(t.m);
            if (idx !== -1) {
                game.createLightning(game.playerPos.x, game.playerPos.y, t.m.x, t.m.y, '#0a84ff');
                game.createExplosion(t.m.x, t.m.y, '#0a84ff');
                
                game.entities.monsters.splice(idx, 1);
                game.state.score += (10 + game.config.stats.scoreBonus);
            }
        });
    }
};
```

#### B. 主逻辑调用 (在 `Game.js`)

```javascript
import { SkillStrategies } from './SkillStrategies.js';

// ...

tryCastUltimate() {
    if (this.state.energy >= this.config.stats.maxEnergy) {
        console.log(`Ultimate Cast! Hero: ${this.config.hero.name}`);
        
        // 1. 通用表现 (音效、能量清空、特效)
        AudioManager.playSfx('skill_' + this.config.hero.id);
        this.state.energy = 0;
        bus.emit('skill-ready', false);
        this.createShockwave(this.playerPos.x, this.playerPos.y, 500, this.config.hero.color);

        // 2. 策略分发 (核心改变)
        const strategy = SkillStrategies[this.config.hero.id];
        if (strategy) {
            // 将 'this' (Game实例) 传递给策略函数
            strategy(this);
        } else {
            console.warn(`No skill strategy defined for hero ${this.config.hero.id}`);
        }
    }
}
```

## 3. 进阶：数据驱动配置

如果技能逻辑非常相似（比如都是造成伤害，只是数值不同），我们可以更进一步，用 JSON 配置驱动策略。

```javascript
const SkillConfig = {
    3: { type: 'aoe_damage', radius: 300, damage: 50 },
    4: { type: 'heal', amount: 100 }
};

const SkillHandlers = {
    'aoe_damage': (game, params) => { /* 通用AOE逻辑，使用 params.radius */ },
    'heal': (game, params) => { /* 通用治疗逻辑 */ }
};

// 调用时
const config = SkillConfig[heroId];
const handler = SkillHandlers[config.type];
handler(game, config);
```

## 4. 总结规范

对于本项目（单词地下城 v2.0），我们应当遵循以下规范：

1.  **拒绝巨型 switch/if-else**：在 `Game.js` 中，任何涉及“根据 heroId 做不同事”的逻辑，一旦超过 3 行，就应该考虑提取为策略。
2.  **提取文件**：新建 `js/core/SkillStrategies.js` 存放所有大招逻辑。
3.  **参数传递**：策略函数接收唯一的 `game` 上下文参数（或者需要的特定子系统），保持函数纯净。
