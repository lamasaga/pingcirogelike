# 多字母拖拽功能开发文档

## 1. 功能概述
为了提升 PC 端用户的操作效率，实现了**框选**和**批量拖拽**功能。允许玩家通过鼠标画框一次性选中多个字母，并将它们作为一个整体移动到新的位置。

## 2. 核心实现逻辑

### 2.1 框选机制 (Box Selection)
*   **触发条件**：在网格区域 (`.grid-section` 或 `.grid-container`) 的**空白处**按下鼠标左键。
*   **视觉实现**：
    *   创建一个 `div.selection-box`（绝对定位），其 `top/left/width/height` 跟随鼠标移动动态计算。
    *   使用 CSS 给被选中的 `.letter-card` 添加 `.selected` 样式（蓝色高亮）。
*   **选中逻辑**：
    *   使用简单的 AABB (Axis-Aligned Bounding Box) 碰撞检测。
    *   遍历所有网格单元 (`.grid-cell`)，判断其矩形区域是否与选择框重叠。
    *   如果重叠且格子内有字母，将其坐标 (`r,c`) 加入 `selectedCells` 集合。

### 2.2 多选拖拽 (Multi-Drag)
*   **触发判断**：
    *   当用户在某个字母上按下鼠标时，系统检查该字母是否位于 `selectedCells` 集合中。
    *   **是**：触发 `startMultiDrag`，移动整个集合。
    *   **否**：清空当前选择，触发普通的单选 `startDrag`。
*   **视觉反馈 (Ghost Elements)**：
    *   不直接移动 DOM 元素，而是创建一组“幽灵元素” (Clone)。
    *   计算每个选中字母相对于**鼠标按下位置**的偏移量 (`offsetX`, `offsetY`)。
    *   拖拽过程中，移动一个容器 (`ghostContainer`)，所有幽灵元素在容器内保持相对位置不变。

### 2.3 放置与冲突检测 (Drop & Collision)
*   **目标计算**：
    *   根据鼠标释放的位置，结合每个字母的相对偏移，计算它们各自的**目标网格坐标**。
*   **原子性原则 (All or Nothing)**：
    *   移动必须全部成功，或者全部失败。
    *   **步骤**：
        1.  先将所有被拖拽的字母从原位置逻辑上“移除”（暂存）。
        2.  检查所有计算出的**目标位置**：
            *   是否超出边界？
            *   是否已被非本次移动的字母占用？
        3.  **无冲突**：将字母写入新位置，刷新 UI。
        4.  **有冲突**：将字母放回原位置（逻辑回滚），不做任何改变。

## 3. 遇到的问题与解决方案

### 问题一：框选触发手感不佳
*   **现象**：鼠标必须精确点击在网格缝隙中才能画框，稍微碰到格子边缘就无法触发。
*   **原因**：最初的事件监听仅局限于 `grid-container` 内部，且对 `target` 的判断过于严格。
*   **解决**：
    *   扩大事件监听范围至 `.grid-section`（包含 padding 的外层容器）。
    *   逻辑修改：只要点击的不是 `.letter-card` 或 `.grid-cell` (即交互元素)，就判定为开始框选。

### 问题二：选中后无法整体拖动
*   **现象**：选中多个字母后，点击其中一个进行拖拽，结果只有被点击的那一个动了，其他选中项丢失。
*   **原因**：
    *   字母卡片上绑定了 `startDrag` 事件。
    *   原有的 `startDrag` 逻辑默认为“单选模式”，点击瞬间会清空之前的选择。
    *   `DragManager` 的全局 `mousedown` 监听被组件级别的 `stopPropogation` 阻断，无法拦截。
*   **解决**：
    *   在 `startDrag` 入口处增加判断：`if (this.selectedCells.has(currentCell))`。
    *   如果点击的是已选中项，直接跳转到 `startMultiDrag` 逻辑，跳过单选初始化。

### 问题三：目标位置计算与回滚
*   **现象**：拖拽一组字母，如果部分目标位置有空位，部分有冲突，会导致“部分移动”，破坏了原来的阵型。
*   **解决**：
    *   引入**预检查**机制。
    *   先计算所有目标位置，只要发现任意一个无效位置，立即终止操作，触发回滚。
    *   确保原位置的数据在确认无冲突前不被覆盖，或者有可靠的恢复手段。

### 问题四：拖拽视觉偏移
*   **现象**：拖拽开始瞬间，幽灵元素的位置与原始元素不能完美重合，有跳变。
*   **解决**：
    *   在 `startMultiDrag` 时，基于 `getBoundingClientRect` 精确计算每个元素相对于鼠标点击点 (`startX`, `startY`) 的差值。
    *   Ghost 渲染时严格使用这个差值进行绝对定位。

## 4. 后续优化方向
*   **触摸屏适配**：目前框选主要针对鼠标操作，触摸屏双指手势或长按进入多选模式尚未实现。
*   **自动避让/交换**：目前采取“遇阻回滚”策略，未来可考虑实现“整体挤开”或“整体交换”的高级逻辑。
